## 2026-05-31 — Исправления: сейв с «битым гарнизоном» больше не вылетает при загрузке и сам чинится, удаление мода из сейва реально убирает его из записи сейва, полосы прокрутки снова кликаются и не налезают на данные, базовые модули игры убраны из списка очистки сейва, анализатор модов больше не ругается на выключенные моды

> Видимая версия мода остаётся `v1.4.0` навсегда.

Подборка исправлений:
- **сейв с «битым гарнизоном» больше не вылетает при загрузке.** Если в сохранении остался гарнизон без своего города или замка (поселение пропало по ходу кампании или из-за смены набора модов), игра вылетала на рабочий стол при каждой попытке загрузить такой сейв — ещё на этапе подсчёта силы кланов. Теперь Crash Doctor не даёт игре упасть в этом месте и при входе в кампанию сам убирает сломанный гарнизон; после того как пересохранишься, сейв станет чистым навсегда. Этот вылет теперь ещё и распознаётся в анализаторе — с объяснением причины и шагами;
- **«Удалить мод из сейва» теперь действительно убирает мод из записи сейва.** Раньше после очистки имя мода (например, русификатора) оставалось в списке модулей сейва, и казалось, что ничего не изменилось. Теперь имена вычищенных модов убираются из записи сейва (и из предупреждения игры «сейв сделан с модом …»), а окно результата честно показывает, сколько объектов реально удалено — и прямо пишет, когда у мода своих данных в сейве не было (перевод или контент-мод);
- **полосы прокрутки снова кликаются мышью и не налезают на данные.** На всех вкладках Crash Doctor (и в окне очистки сейва) полосу прокрутки опять можно тащить мышью, и она больше не перекрывает карточки и списки;
- **базовые модули игры убраны из списка очистки сейва.** Из-за ошибки с регистром в списке «удалить мод из сейва» мог показаться служебный модуль игры (Sandbox) — удаление такого сломало бы сейв. Теперь базовые и служебные модули там не показываются вовсе, плюс добавлена защита, которая не даёт их вычистить, даже если они как-то туда попадут;
- **анализатор модов больше не ругается на выключенные моды.** Раньше на вкладке «Моды» Crash Doctor проверял конфликты патчей, зависимости и порядок загрузки по всем установленным модам подряд — в том числе по выключенным в лаунчере, которые игра вообще не грузит. Из-за этого выключенный мод мог получить красную метку «конфликт» или «нет зависимости», хотя он не работает и навредить не может. Теперь эти проверки идут только по включённым модам; выключенные показываются как «выкл» без ложных меток.

### Подробнее: вылет при загрузке сейва с гарнизоном без поселения

В Bannerlord у каждого гарнизона всегда есть свой город или замок. Если по ходу
игры (или из-за того, что между сессиями поменялся набор модов) поселение пропало,
а сам отряд гарнизона остался в сейве — при загрузке игра, пересчитывая силу всех
кланов, обращается к отсутствующему поселению и **вылетает на рабочий стол**.
Падает каждый раз на одном и том же месте, не случайно. Это вылет в родном коде
игры на испорченных данных сейва, а не ошибка Crash Doctor или TOR.

Crash Doctor теперь страхует это место: вместо вылета загрузка проходит дальше, а
сразу после входа в кампанию мод находит «осиротевший» гарнизон и аккуратно его
убирает. Достаточно один раз **сохраниться** после загрузки — и проблема уходит из
сейва насовсем. Если хочешь идеально чистый сейв без правок — можно вместо этого
загрузить более раннюю сохранёнку, до появления проблемы.

## 2026-05-26 — Исправления: персонажи не лежат на боку, точнее распознаются вылеты видеокарты, нет ложных предупреждений о конфликте модов, карточка очистки кэша видеодрайвера, ложное расхождение сейва из-за версии перевода, сохранность диагностических логов при очистке крашей, понятное сообщение для краша без логов, экспорт прикладывает логи ButterLib

> Видимая версия мода остаётся `v1.4.0` навсегда.

Подборка исправлений:
- **карточка очистки кэша видеодрайвера больше не зависает.** Раньше на больших кэшах NVIDIA (2+ ГБ) карточка «очистить кэш видеодрайвера» оставалась на экране после нажатия «Очистить» и просила нажать снова — хотя файлы держит сам драйвер и удалить их на ходу нельзя. Теперь эта карточка появляется только при реальных признаках сбоя драйвера в логах, а не просто из-за размера кэша;
- **вкладка «Сейвы» больше не пугает ложным расхождением.** Обновление версии мода-перевода (например, русификатора) или мелкое обновление мода больше не помечается как несовместимость — учитываются только изменения версии, которые реально могут сломать загрузку сейва;
- **очистка крашей больше не удаляет файлы, нужные для диагноза.** Раньше при очистке старых крашей (с сохранением дампов) из папок пропадали логи движка, и потом такой краш нельзя было распознать. Теперь логи краша при очистке сохраняются всегда;
- **краш без логов теперь объясняется по-человечески.** Если игра не успела записать логи (или они уже ушли через встроенный репортёр игры), Crash Doctor пишет «в этой папке краша нет данных движка, анализировать нечего» — и не просит присылать пустой архив;
- **экспорт краша стал полезнее.** К архиву теперь прикладываются логи ButterLib — в них подробная запись ошибки, по которой видно, какой именно мод вызвал вылет. Это заметно ускоряет разбор крашей, которые раньше выглядели как «непонятный вылет»;
- **персонажи больше не «лежат на боку».** Если в настройках Crash Doctor по ошибке включить защиту от ошибок для базового модуля игры (например, Sandbox), у всех персонажей пропадала анимация и модели рисовались в неестественной позе на боку. Теперь служебные модули игры в списке защиты не показываются вовсе, а если такая галка уже была включена раньше — мод снимает её сам при запуске;
- **точнее распознаются вылеты видеокарты.** Добавлено распознавание вылета по сообщению движка о потере видеокарты (lock_texture / GetDeviceRemovedReason) и отдельный разбор частого случая на картах с 4 ГБ памяти, когда кэш шейдеров так и не собран и игра пересобирает шейдеры каждый запуск — с пошаговым советом, что сделать;
- **пропали ложные предупреждения о конфликте модов.** Сам Crash Doctor и служебные моды (Harmony, ButterLib, MCM и подобные) больше не помечаются значком «конфликт патчей» — раньше они зря пугали игроков, хотя реального конфликта не было.

## 2026-05-22 — Защита от вылетов на ходу AI, при завершении заданий и при загрузке сейва, привязка к мощной видеокарте на лэптопах, pagefile уважает ручные split-конфигурации и не убирает всё с диска C, распухание героев по шаблону + бэкап Ironman-сейвов, умнее советы при крахе на ландшафте, отдельная карточка под регрессии драйвера AMD RX 9000, новое правило про вылет на «Назад» в создании персонажа

> Видимая версия мода остаётся `v1.4.0` навсегда.

Несколько правок одновременно:
- **новый защитный слой против вылетов на ходу AI и при завершении заданий**: Crash Doctor перехватывает четыре типичных вылета (подсчёт жалованья / провизии / часовой такт AI / завершение задания альтернативным путём) когда в отряде остаётся «потерянный» юнит после другого мода, и показывает игроку всплывающее сообщение что вылет предотвращён;
- **новая чистка сейва при загрузке**: при входе в кампанию Crash Doctor проходит по всем отрядам, поселениям и активным заданиям и убирает сломанные записи о бойцах (юнитов, которых уже нет в игре), пока следующий час на карте или смена дня не выкинули игру с NRE;
- два новых тумблера в разделе **Настройки** Crash Doctor (по умолчанию **ВКЛ**) — можно выключить любую из двух защит, если она конфликтует с другим модом;
- **прокрутка в разделе «Настройки»** — все карточки с настройками теперь скроллятся, на низких разрешениях карточки больше не уходят за пределы экрана;
- новая Tune-Up карточка для лэптопов с гибридной графикой (Intel + NVIDIA / AMD) — Crash Doctor одной кнопкой прибивает Bannerlord к мощной видеокарте;
- pagefile-карточка теперь видит ручные split-конфигурации игрока (маленький pagefile на C: + большой на D:) и не настаивает на пересборке, плюс собственная Apply-логика оставляет маленькую запись на C: вместо того чтобы переносить всё на другой диск (для дампов памяти ядра);
- карточка «Освободить место на дисках» опускает порог C: с 50 ГБ до 20 ГБ, если pagefile уже стоит на другом диске — больше не требует освободить место под рост pagefile, которого там и не будет;
- Save Health-карточка про распухание героев по шаблону + чистка, и безопасная очистка сейва в режиме Ironman;
- два уточнения по крашам на отрисовке ландшафта (Missing shader from sack: pbr_terrain): отдельный диагноз про регрессию драйвера AMD на RX 9000 / RDNA 4 + разделение совета «пересобрать шейдеры» на «кэш не собирали» vs «sack-файл повреждён»;
- карточка «пересобрать кэш шейдеров» больше не вылезает на безобидные настройки (разрешение, FXAA, постобработка и т.п.);
- новое правило распознавания — вылет при нажатии «Назад» на экране создания персонажа (типичный конфликт TOR-карьеры с другим модом, который патчит тот же экран).

### Новое: защита от вылетов на ходу AI и при завершении заданий

В кампании каждый час игрового времени AI считает жалованье и провизию для
всех отрядов на карте и принимает решения куда им двигаться. Раз в сутки
игра завершает активные задания, которые игрок отправил решать «альтернативным
путём» (отправил отряд в гарнизон лорду, на засаду каравана и т.п.). Если
**другой мод** удалил какого-то воина не до конца — оставил «потерянную»
запись о нём в списке отряда или в отправленном на задание подкреплении —
на следующем такте игра пытается обратиться к несуществующему персонажу и
**вылетает на рабочий стол**. Симптом в краш-логе: NullReferenceException
где-то внутри `TORPartyWageModel.CalculateCharacterWage`,
`TORMobilePartyFoodConsumption`, `AiPatrollingBehavior.AiHourlyTick` или
`IssueBase.AlternativeSolutionEndWithSuccess` под `IssueManager.DailyTick`.

Crash Doctor теперь оборачивает все четыре места защитой. Когда срабатывает:
- Для часовых тиков AI: проблемный отряд **пропускает один час** на карте
  кампании (без жалованья и решения куда идти на этом такте), и игра идёт
  дальше.
- Для завершения задания: до того как игра дёрнет сломанный список,
  Crash Doctor **тихо вычищает «потерянных» бойцов** из отправленного
  отряда — задание завершается штатно. Если что-то всё-таки проскочило —
  тик дня выживает, задание попробует завершиться ещё раз через сутки
  уже на чистом списке.
- В правый-нижний угол экрана выскакивает **янтарная плашка** —
  «Crash Doctor: предотвращён вылет при подсчёте жалованья отряда. Игра
  продолжается.» (или аналогичная для завершения задания) — игрок видит
  что мод сработал.
- Подробности (имя отряда, лидер, размер списка, ID задания, что именно
  сломалось) пишутся в `crashdoctor.log` — этого хватает чтобы автору
  мода-виновника завести баг-репорт.

Защита **общая** — работает в любом моде, не только в The Old Realms. Внутри
TOR ловит самый частый сценарий: чужой мод порождает временного героя,
помещает его в чужой отряд (как лидер караван-эскорта, как кандидат на найм
и т.п.) и потом удаляет героя, забывая выгрести из отрядов.

Тумблер **«Защита от вылетов на жалованье / провизии / AI»** в разделе
**Настройки** Crash Doctor — по умолчанию **ВКЛ**, можно выключить если она
конфликтует с другим модом который тоже патчит те же места.

### Новое: чистка сейва от сломанных бойцов при загрузке

То же явление что выше («потерянный» юнит в отряде после другого мода),
но **до** того как игра успеет вылететь. При входе в кампанию Crash Doctor
проходит по всем отрядам, поселениям **и активным заданиям** (юниты,
которых игрок отправил решать issue альтернативным путём) на карте и
удаляет записи о бойцах, у которых уже **нет реального персонажа** в игре
(`Character == null` или у персонажа пропала культура).

Если что-то нашлось и почистилось — в правый-нижний угол выскакивает
**зелёная плашка**: «Crash Doctor: очищено N повреждённых записей в
отрядах — сохранение защищено от возможного вылета.» Если всё чисто —
ничего не показывает и не пишет в лог.

Зачем нужно отдельно от защиты на ходу AI: некоторые сейвы накопили
столько мусора, что вылет случается ещё до первого часового такта AI —
например при попытке игры посчитать общую силу отряда сразу после
загрузки. Чистка до старта тиков убирает эту волну.

Тумблер **«Чистить сломанные отряды при загрузке»** в **Настройках** —
по умолчанию **ВКЛ**. Изменения вступают в силу при следующей загрузке
сейва, перезапуск игры не нужен.

### Улучшено: прокрутка в разделе «Настройки» Crash Doctor

С появлением двух новых тумблеров в разделе **Настройки** общее число
карточек выросло до шести (плюс блок «Глотать ошибки конкретных модов»
с прокручиваемым списком модов). На стандартных и низких разрешениях
последние карточки уходили за нижний край окна.

Теперь карточки настроек вертикально **прокручиваются** — справа появилась
полоса прокрутки, как на вкладке «Настройка системы». Заголовок раздела
остаётся неподвижным сверху. Каждая карточка по-прежнему сохраняет свой
полный размер и подсказку, ничего не урезано.

### Новое: правило про вылет при нажатии «Назад» на экране создания персонажа

В правила распознавания добавлено `tor.character_creation_back_crash` — ловит
вылет на старте новой кампании когда игрок жмёт кнопку **«Назад»** на
экране создания персонажа TOR-карьеры. Симптом — `NullReferenceException`
где-то внутри `CharacterCreationContent`, и нет очевидной зацепки в стеке
у какого мода виновата эта кнопка.

Карточка-диагноз объясняет что обычно виноват чужой мод, патчащий экран
создания персонажа (несовместимая правка карьер / культур / архетипов), и
советует временно убрать чужие career-моды поверх TOR чтобы попасть в игру.
В большинстве случаев одного запуска без проблемного мода хватает чтобы
посмотреть про какой именно мод идёт речь — после этого игрок включает его
обратно и продолжает играть, проходя экран создания персонажа без нажатия
«Назад».

### Новое: привязка Bannerlord к мощной видеокарте на лэптопах (Tune-Up карточка)

В **Настройка системы** появляется новая карточка когда:
1. На компьютере найдены **сразу две видеокарты**: встроенная (Intel UHD /
   Iris, AMD Vega / Radeon Graphics) и дискретная (NVIDIA GeForce / RTX /
   GTX, AMD Radeon RX / Pro, Intel Arc).
2. В **последних краш-репортах** видно, что картинка шла через разъём
   встроенной видеокарты, пока дискретная простаивала, **либо**
   компиляция шейдеров террейна спамила «Missing shader from sack:
   pbr_terrain» (классический симптом старого Intel UHD-драйвера).

«Применить» дописывает одну запись в реестр пользователя — Windows будет
запускать `Bannerlord.exe` на **высокопроизводительной видеокарте**. Это
та же самая кнопка, что в **Параметры → Дисплей → Графика → Bannerlord →
Высокая производительность**, только не нужно искать exe-файл вручную.

Зачем: на лэптопах с двумя видеокартами выход монитора часто заведён
через встроенную графику, и без явной привязки игра рендерится на ней —
даже если рядом стоит RTX 3050 или RX 6700M. Встроенная не тянет шейдеры
террейна The Old Realms; в самом начале большой битвы или штурма
получается вылет по нехватке видеопамяти + битый сейв (последний краш
ловит игру в момент записи сейва).

Без прав администратора, без перезагрузки Windows — нужен только
перезапуск самого Bannerlord, чтобы новая видеокарта вступила в силу.
Если ничего не изменилось, в BIOS лэптопа может быть жёсткий переключатель
графики (MUX) или режим «только дискретная», который перекрывает
настройку Windows. Карточка откатываемая: rollback вернёт прежнее
значение реестра.

Карточка не показывается на стационарных ПК с одной видеокартой и на
лэптопах, где монитор и так заведён через дискретную — мы не подсовываем
«полезные советы на всякий случай».

### Исправлено: Crash Doctor теперь видит ручной pagefile на другом диске

Карточка «Увеличить pagefile до 40 / 60 ГБ для TOR» читала только
**первую** запись в реестровом массиве `PagingFiles`. Если игрок
вручную (через свойства системы Windows) сделал split — маленький
pagefile на C: + большой на D: — карточка видела только C-шный
кусочек, считала что pagefile «4096 / 4096 МБ», и продолжала
требовать пересборки, хотя на D: уже стояли нужные 60–80 ГБ.

Карточка «Освободить место на дисках» жёстко требовала **50 ГБ
свободно на C:** для любого сценария. Тоже несправедливо: 50 ГБ
нужны на C: только если pagefile растёт там же. Когда pagefile
уже уехал на другой диск (≥30 ГБ max), на C: достаточно 20 ГБ
для OS-headroom и временных файлов компиляции шейдеров.

Теперь:
- M11 (pagefile) проходит по **всем** записям `PagingFiles`,
  берёт ту, у которой наибольший max. Если хоть одна запись
  набирает 40 000 / 60 000 МБ — карточка становится Healthy и
  больше не лезет. В тексте теперь видно конкретный диск:
  «Pagefile уже настроен под TOR (60000/80000 МБ на D:)».
- M13 (свободное место) проверяет, есть ли «большой» pagefile
  (≥30 ГБ max) на не-C: диске. Если есть — порог свободного
  места на C: опускается с **50 ГБ до 20 ГБ**, и в evidence
  попадает строка «Pagefile is on D: — C: threshold lowered to
  20 GB».

Тот случай, когда игрок «всё уже сделал», но карточки продолжали
зудеть, больше не повторяется.

### Исправлено: настройка pagefile больше не убирает его с диска C

Раньше Tune-Up карточка «Увеличить pagefile до 40 / 60 ГБ для TOR»
работала по принципу: если на C: меньше 65 ГБ свободно, она выбирала
самый свободный другой диск (обычно D: или E:) и **перезаписывала весь
массив `PagingFiles` в реестре одной записью на этот диск**. В итоге
у Windows вообще не оставалось pagefile на системном диске.

Чем это плохо: без pagefile на C: ОС не может писать kernel memory dump
при BSOD (memory.dmp остаётся пустой), отказывает hybrid sleep, и часть
системных компонентов теряет резерв commit-памяти на загрузке.

Теперь логика умнее:

- **C: ≥ 65 ГБ свободно** → pagefile целиком на C: (стандартный
  Windows-вариант, как раньше).
- **C: 8–65 ГБ свободно И есть другой диск с ≥ 65 ГБ** → **сплит**:
  маленький pagefile **800 / 4096 МБ остаётся на C:** (хватает на
  kernel memory dump на BSOD), а большой TOR-овский **40000 / 60000 МБ
  уходит на свободный диск**. C: pagefile больше никогда не удаляется
  молча.
- **C: < 8 ГБ свободно** → fallback на solo на другом диске (как
  раньше), но в этом случае предупреждение про дампы памяти.

Если уже применил старый вариант и pagefile пропал с C: — нажми
«Откатить» в Журнале Crash Doctor → перезагрузка → дальше можно
применить заново уже с правильным split-поведением.

### Новое: распухание героев по шаблону (Save Health карточка + чистка)

В **Save Health** появляется новая карточка, когда у одного и того же
шаблона персонажа (`CharacterObject`) в живых **больше 100 экземпляров**.
Это поздне-кампанийный паттерн от модов, у которых `HourlyTickParty`
без ограничения создаёт новых героев — ярче всего видно на Greenskin-патчах
LoreHardcore, но проблема не привязана к конкретному моду.

Почему важно: каждый Hero тянет за собой свой `Equipment`, который
грузит и удерживает уникальный набор GPU-текстур. Когда shader cache
больше не помещает их все, игра начинает падать с `0xC0000005` внутри
рендера — обычно через секунду после перемотки на x4–x8. Сам сейв
грузится нормально, ущерб копится во время игры. У игроков, которые
сняли мод-источник и продолжили на том же сейве, вылеты прекращались —
это и подтверждает диагноз.

Что делает чистка (при нажатии **Clean** в панели Save Health): по
каждому шаблону выше порога оставляет первые **20 героев нетронутыми**
(чтобы тип бойца продолжал спавниться), а остальных помечает к удалению
**только** если они проходят строгие проверки — не лидер клана, не
нотабль, не лидер партии, не пленник, не привязан к поселению / посту
губернатора, нет семейных связей (супруг / родитель / ребёнок), не
компаньон / странник. Save Cleaner затем обнуляет все ссылки и пишет
отдельный `CrashDoctor_cleaned_<дата>.sav`. Исходный сейв на диске не
трогается.

Карточка не показывается на здоровых сейвах: NotApplicable если
кампания не загружена, Healthy если у всех шаблонов ≤100 живых героев.
Симптом в краш-логе для подтверждения диагноза: `0xC0000005` плюс
спам `Missing shader from sack: pbr_terrain` в `rgl_log_*.txt`, как
правило после быстрой перемотки.

### Исправлено: очистка сейва в режиме Ironman теперь безопасна

В режиме Ironman у Bannerlord один-единственный слот сохранения, и
движок игнорирует имя, переданное в `SaveAs` — любое сохранение
перезаписывает тот же файл `Ironman<UniqueGameId>.sav`. Поэтому
обычный путь Save Cleaner («исходный .sav никогда не перезаписывается,
пишем новый `CrashDoctor_cleaned_*`») в Ironman не работает: движок
всё равно затирает оригинал.

Теперь Save Cleaner это распознаёт: до начала чистки делает копию
исходного `Ironman<UniqueGameId>.sav` рядом, с расширением `.bak`.
Если чистка пошла не так — оригинал можно вернуть, удалив затёртый
`.sav` и переименовав `.bak` обратно. В UI после чистки выводится
честное сообщение «сейв перезаписан, бэкап лежит здесь», а не
обычное «создан новый файл».

### Новое: распознаём регрессию драйвера AMD на RX 9000-серии (RDNA 4)

После того как видеокарты Radeon RX 9000-серии (RDNA 4 — RX 9060,
RX 9070, RX 9070 XT) попали в массовый продакшен, AMD за весну
2026 выпустила несколько драйверов с **нерешёнными регрессиями
рендера**, перечисленными в их же официальных release notes:
артефакты в God of War, креши и driver-таймауты в RoadCraft,
артефакты в Satisfactory, креши при multi-monitor + FreeSync. Тот
же шейдерный компилятор драйвера собирает sack-шейдеры Bannerlord
на лету; на RDNA 4 с этими драйверами часть TOR-перестановок
`pbr_terrain` либо не компилируется, либо ложится в кэш битой.

Симптомы у игрока:
- В rgl_log спам `Missing shader from sack: pbr_terrain` и краш
  `0xC0000005` в D3D11 на отрисовке ландшафта (то же что у
  обычного битого sack-файла).
- **Переустановка игры не помогает** (файлы игры целы).
- **«Build Shader Cache» не помогает** даже после нескольких
  попыток (компилятор тот же, результат тот же).
- Активная видеокарта — Radeon из серии RX 9000, драйвер от
  **2026-03-01 или новее**.

Crash Doctor теперь поднимает отдельную карточку с первой
рекомендацией: **откатить драйвер на AMD Adrenalin 26.3.1 WHQL
через DDU в Safe Mode**. Это самый надёжный фикс по репортам
сообщества по нескольким играм, не только Bannerlord. В описании
карточки расписаны промежуточные шаги (выключить HYPR-RX /
Anti-Lag / AFMF / FreeSync), путь обновления внутри 26.5.x
(26.5.1 → 26.5.2), и причина почему Win 10 EOL может усугублять
ситуацию (AMD приоритизирует QA свежих карт под Win 11).

Карточка не показывается на стационарных ПК с не-AMD картой, на
старых Radeon (RX 5000 / 6000 / 7000), на свежих RX 9000 с
драйвером **старше марта 2026** (которые в эту волну регрессий
не попали), или если в логе нет паттерна `Missing shader from
sack: pbr_terrain`. Параллельно с ней может показываться и общая
карточка про повреждённый sack — это не противоречие, обе
рекомендации валидны, RDNA4-специфичная просто более точная.

### Исправлено: совет «пересобрать шейдеры» при крахе на ландшафте теперь не повторяется впустую

Когда игра падала на отрисовке тайла ландшафта (типичный лог: десятки строк
«Missing shader from sack: pbr_terrain» подряд → access violation в D3D11),
Crash Doctor показывал одну и ту же рекомендацию — нажать «Build Shader
Cache» в главном меню. У большинства игроков это срабатывало с первой
попытки. Но если у игрока битый sack-файл (Steam скачал криво, антивирус
выкусил часть файла, мод из Workshop повредился) — пересборка кэша ничего
не давала: «Build Shader Cache» собирает пользовательский кэш **из** этих
же sack-файлов, поэтому из битого исходника получается битый кэш сколько
ни пересобирай.

Crash Doctor теперь различает два случая по логу:

- **Кэш ни разу не собирали** (в логе нет строки про чтение
  `read_compressed_shader_cache_package`) — рекомендация: «Build Shader
  Cache» в главном меню. Это тот случай, который чинится с первого раза.
- **Кэш собирали, но sack-файл повреждён** (строка про чтение есть,
  но шейдеров внутри нет) — рекомендация перевёрнута: сначала **Steam →
  Свойства игры → Установленные файлы → Проверить целостность**, потом
  переподписаться на TOR_Core / TOR_Environment / TOR_Armory в Workshop
  (контент часто докачивается частично), и только потом контрольный
  «Build Shader Cache». Для свежих видеокарт (RTX 50-серия, RX 9000-серия)
  отдельно отмечен откат драйвера через DDU — самые новые драйверы иногда
  не могут скомпилить отдельные TOR-перестановки и кэшируют провал в sack.

Зачем разделение: игрок, у которого битый sack, до этого пересобирал
шейдеры по 4–5 раз и упирался в один и тот же краш. Теперь правильный
шаг — Steam Verify — показывается **первым** и заметным пунктом, а не
третьим в общем списке.

### Исправлено: карточка «пересобрать кэш шейдеров» больше не вылезает на безобидные настройки

Раньше карточка «Графика изменилась — очисти кэш шейдеров перед следующим
запуском» появлялась после **любого** изменения почти 40 разных
графических ключей: смена разрешения, переключение оконного режима,
включение dynamic resolution, переключение FXAA → TAA, любая постобработка
(bloom / vignette / depth of field / motion blur / chromatic aberration
и т.д.), смена анизотропной фильтрации текстур, выбор DLSS-режима,
изменение детализации персонажей или окружения.

Проблема: ни одна из этих настроек **не инвалидирует** скомпилированные
шейдеры — все они применяются движком на лету (через runtime-параметры
сцены / per-view постобработку / SwapChain). А карточка просила
перезагрузить ПК и потратить 20–60 минут на пересборку **впустую**.

Сейчас карточка реагирует только на 9 настроек, у которых смена реально
может потребовать пересборки: качество шейдеров, террейна, теней
(фильтрация + тип), листвы, воды, освещения, тесселяции, общее качество
текстур. Если изменил что-то другое (например, разрешение монитора с
1920×1080 на 2560×1440 или включил dynamic resolution) — Crash Doctor
больше не предлагает пересборку, потому что она там не нужна.

### Новое: встроенная очистка сейвов

Раньше при «раздутом» сейве (50+ МБ, orphan-партии от удалённых модов)
Crash Doctor мог только посоветовать поставить сторонний **Save Cleaner**
с Nexus. Теперь чистка работает **прямо в моде** — ничего ставить
дополнительно не нужно.

Что чистится:
- **Заброшенные крафченые предметы** — потерявшие хозяина после удаления модов крафта.
- **«Глючные» партии** — мобильные отряды без `PartyComponent` или без действующей фракции (типичный артефакт удалённого мода).
- **Битые записи в журнале логов** — записи событий с null-ссылками на героев/поселений.
- **Удаление модов из сейва** — выбираешь моды в списке, мод тоже очищает все объекты, принадлежащие выбранным модам, разом в одном проходе.

Алгоритм консервативный: исходный `.sav` **никогда не перезаписывается** —
мод всегда создаёт новый файл `CrashDoctor_cleaned_<дата>.sav`. Если что-то
пошло не так — оригинал остаётся целым, перезагрузишь его и продолжишь как
раньше.

Адаптировано из **JungleDruid / bannerlord-save-cleaner** (лицензия MIT,
текст лицензии включён в моде: `Saves/Cleaner/LICENSE-SaveCleaner.txt`).

### Новое: вкладка «Моды»

Пятая вкладка в главном меню Crash Doctor. Анализирует **все установленные
модули** (не только активные) и показывает:

- **Проблемы загрузки**: отсутствующие зависимости, несоответствие версий,
  заблокированные DLL (Windows MOTW), цикл в графе зависимостей.
- **Безопасность сейвов**: моды, которые пишут в сейв через
  `SaveableTypeDefiner` — их нельзя добавлять/убирать на ходу.
- **Тип модуля**: чистый контент (XML/ассеты), код через Harmony, BLSE
  плагин и т.д.
- **Размер**: чтобы видеть «толстые» моды.

Каждый ряд кликабелен — попап показывает полный отчёт: какие DLL, какие
зависимости, что предупреждает анализатор, ссылку на Workshop / Nexus
(если нашли).

### Новое: горячая клавиша Ctrl+D — анализ сейва из кампании

Раньше для запуска диагностики сейва нужно было выходить в главное меню.
Теперь — `Ctrl+D` прямо в карте/городе/бою:

- Если кампания загружена → открывается **отдельный экран анализа сейва**:

- Если в главном меню → открывается обычный экран Crash Doctor.

Удобно когда сейв стал крашиться или тормозить — не надо выходить, чтобы
проверить «а что у меня в сейве».

### Новое: опциональные защитные модули

В **Настройках** Crash Doctor появились три тогла защиты. Все **выключены
по умолчанию** — включаешь только если столкнулся с проблемой:

- **«Перехватывать exception в UI»** — оборачивает методы UI (инвентарь,
  энциклопедия, экраны партий) Harmony-финализатором. Если сторонний мод
  ломает UI exception'ом — игра не падает, а показывает красную надпись.
  Только UI-методы, не трогает игровую логику.
- **«Подавлять exception мода»** — список модов, чьи исключения внутри
  mission/campaign-логики молча проглатываются. Спасает от падений из-за
  одного известного бажного мода без выкидывания всего, что от него
  зависит.
- **«Логировать нативные краши»** — устанавливает Windows VEH (Vectored
  Exception Handler). Когда игра падает на native AV / heap corruption /
  stack overflow — в `crashdoctor.log` будет timestamp + код исключения
  ещё до того, как TaleWorlds покажет свой crash dialog. Без этого нативные
  падения видны только в `.dmp` файле.

Плюс встроенные (без тогла):
- **Аварийное сохранение** — при unhandled exception мод пытается сохранить
  сейв в `CrashDoctor_Emergency_<timestamp>.sav` прежде чем игра умрёт.
  Часто спасает 2-3 часа прогресса.
- **Ограничение скорости в late-game** — после первого native AV в текущей
  сессии или при числе партий ≥ 800 мод запрещает x8 скорость (известная
  TaleWorlds-гонка между HourlyTick-фазами падает с NULL deref на x8 в
  late-game). x4 продолжает работать.
- **Проверка patch-конфликтов** — при запуске self-test проверяет, не
  перебиты ли наши Harmony patches другим модом. Если перебиты — кнопка
  Crash Doctor в главном меню остаётся доступной, но в tooltip пишет
  причину.

### Новое: перенос Windows TEMP на свободный диск (Tune-Up карточка)

В **Настройка системы** появляется новая карточка когда:
1. На системном диске (где лежит `%TEMP%`) свободно **меньше 40 ГБ**.
2. На другом фиксированном диске свободно **больше 100 ГБ**.

«Применить» создаёт `<БольшойДиск>:\Temp\<имя пользователя>` и меняет
переменные среды `TMP` и `TEMP` для текущего пользователя на новый путь.
Системный TEMP (общий для всех пользователей) **не трогается** — права
админа не требуются.

Зачем: компилятор шейдеров Bannerlord пишет промежуточные файлы в
`%TEMP%` во время «Build Shader Cache» и runtime-перекомпиляции. На
зажатом системном диске это валится с **out of disk space** в виде
«pdb append failed» / «debug info append failed» — выглядит как баг
шейдеров, а на деле просто TEMP забит.

После «Применить» **обязательно выйти из Windows и зайти заново** (или
перезагрузка) — переменные среды применяются только в новой пользовательской
сессии. Bannerlord, запущенный в текущей сессии, продолжит использовать
старый TEMP. Откатываемо: rollback вернёт прежнее значение.

## 2026-05-10 — Новая вкладка «Сейвы», правила late-game крашей, удаление модуля оптимизации

> Видимая версия мода остаётся `v1.4.0` навсегда.

**Новое: вкладка «Сейвы».**

Crash Doctor теперь умеет диагностировать сохранения **до загрузки в игру** —
просто читает заголовок файла `.sav`, не запуская кампанию. Появляется в
главном меню как четвёртая вкладка рядом с «Диагностика крашей», «Настройка
системы» и «Журнал».

На карточке каждого сейва видно: имя, дату, размер, день кампании, уровень
героя, золото, размер отряда. Дальше идут расхождения по модам — что было
записано в сейве и что у тебя сейчас активно:

- **«Установлены, но не активны»** — мод стоит, просто галка снята в лаунчере.
  Кнопка одним кликом включает их.
- **«Отсутствуют в системе»** — мод записан в сейве, но его вообще нет в
  лаунчере. Кнопка копирует список IDs в буфер обмена — найти их в Steam
  Workshop / Nexus.
- **«Активны, но не были в сейве»** — добавлены к существующей кампании.
  Часть из них безопасно (Harmony, ButterLib, MCM — добавляются в любую
  кампанию), часть нет — для известных проблемных модов
  (CalradianClans, BannerKings) выводим предупреждение «не добавляется в
  существующую кампанию, начни новую».
- **«Разные версии»** — версия мода в сейве отличается от установленной.

Плюс несколько живых эвристик:

- **Размер сейва ≥ 50 МБ** — почти всегда значит «отряды-призраки» от
  удалённых модов; рекомендуем поставить **Save Cleaner** (Nexus #7763).
- **Несовпадение мажор-версии Bannerlord** — сейв из 1.2.x в 1.3.x и
  наоборот не совместимы.
- **Iron Man** — всегда советует сделать backup перед загрузкой (упадёт —
  файл удалится).
- **Известные мод-зависимости** — если в сейве записан мод, который пишет
  данные в сейв через `SaveableTypeDefiner` (PlayerSettlement, BannerKings,
  TOR_Core, Dramalord, Diplomacy, ImprovedGarrisons, CalradianClans), и
  его сейчас нет — выдаём ссылку, где его заново скачать.
- **Поздняя кампания (день ≥ 700) с размером сейва ≥ 25 МБ** — известный
  риск crash'а от мёртвых ссылок на кланы (KingdomDecision NRE).

Кнопки на карточке: включить отсутствующие моды, выключить лишние,
скопировать IDs, открыть Save System Fix на Nexus (#1925) если есть
полностью отсутствующие моды, открыть Save Cleaner на Nexus (#7763) если
сейв раздут, сделать `.bak`, открыть в проводнике, удалить (в Корзину).

## 2026-05-09 — Окно оптимизации, новые правила крашей, чистка ложных предупреждений

> Видимая версия мода остаётся `v1.4.0` навсегда.

**Новое: окно оптимизации в игре + вкладка Performance в Crash Doctor.**

Опциональная фича для тех у кого FPS падает на больших мод-сборках (TOR,
EE1700) или просто когда на карте много отрядов. По умолчанию **выключено** —
включаешь сам, мод тебя ничем не насилует.

- На карте кампании появляется небольшое окно в правом верхнем углу со
  счётчиками, FPS и сравнением «с оптимизацией / без».
- `Ctrl+P` — включить/выключить оптимизацию мгновенно, без перезапуска игры.
- `Ctrl+O` — показать/скрыть окно. Подсказка про обе клавиши прямо в окне.
- Твой выбор сохраняется в `Documents\Mount and Blade II Bannerlord\CrashDoctor\state\`
  и переживает любые будущие обновления мода.
- Работает на ванильном Bannerlord и любом моде, не только на The Old Realms.

Что именно делает оптимизация когда включена:
- **Далёкие отряды:** не считает почасовое обновление невидимых далёких
  отрядов (даёт самый большой прирост FPS).
- **Города и замки:** реже обновляет далёкие — деревни обновляются как
  раньше, продовольствие в порядке.
- **ИИ далёких отрядов:** думает 2 раза в секунду вместо каждого кадра.
- **Анимации далёких отрядов:** реже обновляются (по умолчанию ВЫКЛ —
  может слегка мелькать).

Сравнение FPS «с оптимизацией / без» показывает разницу только после
**5 минут реальной игры в каждом режиме**. До этого пишет «Замер: ещё
4 мин в ВКЛ + 2 мин в ВЫКЛ». Это сделано чтобы не получать «оптимизация
вредит, −11 FPS» через 12 секунд после Ctrl+P, когда видеокарта только
прогревается.

Окно меняет язык вместе с игрой — переключил RU/EN в настройках
Bannerlord, окно перерисовалось через пару секунд.

**Требует мод `Bannerlord.Harmony`** (бесплатный, в Steam Workshop). Без
него Crash Doctor грузится как обычно; во вкладке Performance будет кнопка
для установки Harmony.

---

**Больше не показывается CAUTION при каждом запуске.**

Лаунчер ругался: «Crash Doctor зависит от Bannerlord.Harmony v2.4.0.0, у тебя
v2.4.2.0 — ты уверен?». Это происходило при каждом обновлении Harmony,
потому что лаунчер сверяет версию очень строго (даже разный последний номер
триггерит предупреждение). Теперь привязка к Harmony перенесена в раздел,
который видят BLSE / Vortex, но не видит ванильный лаунчер. Никаких CAUTION,
порядок загрузки моду по-прежнему правильный.

**Новые правила распознавания крашей:**

- **Утечка видеокарты при долгой непрерывной игре.** На картах NVIDIA
  RTX 50-серии (Blackwell) и AMD RX 9000 (RDNA 4) драйвер постепенно течёт
  памятью; через 3+ часа начинаются падения с ошибкой `create_texture_array`.
  Crash Doctor теперь распознаёт эту связку (флаг GPU Crash + время игры
  больше 3 часов) и советует: выходить из игры на рабочий стол каждые 2-3
  часа (не в главное меню — именно полностью), обновить драйвер, следить за
  расходом видеопамяти в Диспетчере задач.
- Старое правило про `CreateTexture2D` расширено третьей причиной — той же
  утечкой на RTX 50-серии — с конкретными настройками для NVIDIA: панель
  управления → Bannerlord.exe → выключить DLSS Frame Generation, поставить
  «Prefer maximum performance», поставить Game Ready 580.x+.
- Движок правил научился сравнивать «больше чем» / «меньше чем» в значениях
  crash-логов и правильно читать дробные числа на русской Windows (где
  десятичная запятая, а не точка).

**Определение видеокарты на ноутбуках с двумя GPU.**

На игровых ноутбуках (Lenovo Legion, ASUS ROG, MSI, Razer) у дискретной
карты RTX 4060 Laptop / 4070 Laptop и подобных Windows показывает только
3 ГБ видеопамяти вместо реальных 8-12 ГБ. Это известный баг Windows, не
реальный дефицит памяти. Раньше Crash Doctor пугался и писал «мало VRAM,
играть нельзя» — теперь автоматически определяет ноут с двумя видеокартами
и не пугается. На стационарных ПК поведение прежнее.

**Чистка вкладки «Настройка системы»:**

- **Чистка временных файлов:** плашка больше не висит когда чистить
  нечего. Раньше она показывалась всегда когда в `%TEMP%` больше 1 ГБ —
  даже если все файлы свежие и ничего удалить нельзя. Теперь появляется
  только когда есть реально старый мусор (200+ МБ файлов старше 7 дней).
  Если нажал «Применить» а все старые файлы оказались заняты другими
  программами — плашка прячется до перезапуска игры.
- **Windows ждёт перезагрузку:** больше не появляется через 30 секунд
  после включения ПК. Crash Doctor смотрит сколько работает Windows — если
  меньше 10 минут, считает что это остатки прошлой загрузки, а не реальная
  отложенная перезагрузка. Также убран один из сигналов — он почти всегда
  «горит» на здоровой системе из-за рутинной работы Windows.
- **Очистка кэша шейдеров:** теперь умеет снимать «только чтение»
  с папок Steam Workshop. Steam иногда помечает скачанные файлы как
  read-only — при удалении выскакивало «отказано в доступе». Теперь если
  в логах Bannerlord видно много ошибок про шейдеры, Crash Doctor сам
  снимает эту метку перед удалением.

**Счётчик «Найдено N проблем» теперь показывает правду.**

Раньше под надписью «Найдено 3 проблемы» список бывал пустой — потому что
счётчик считал и скрытые карточки (которые юзер раньше пометил как
«Игнорировать»). Теперь число в надписи всегда совпадает с тем что видно
на экране.

**Зависшая плашка прогресса при нажатии «Применить» с внешним приложением.**

Жалоба: нажал «Применить» на карточке «закрыть программы, жрущие память» —
открывается Диспетчер задач, игра уходит на задний план, возвращаешься в
игру а плашка прогресса висит и не закрывается. Двойная защита:

- 6 карточек, чей «Применить» открывает Диспетчер задач / Очистку диска /
  браузер / Проводник, переведены на быстрый путь — плашка прогресса
  для них **вообще не показывается**.
- Дополнительно: если плашка по какой-то причине застряла больше чем на
  30 секунд — она автоматически сама закрывается. Защита от любых будущих
  ошибок такого рода.

**Сэкономлено CPU вместо непонятного «99%».**

Окно оптимизации показывает «Сэкономлено CPU: 12 345 раз» — столько
лишних расчётов мод убрал. Раньше там было «Сэкономлено: 99%» — % от
чего, никто не понимал.

## v1.4.0 — Tune-Up Phase 2: 11 new modules + bundle export + Ignore (2026-05-04)

**Новые модули ремедиации (10):** `M1.2 RAM check`, `M1.5 TEMP cleanup`, `M2.3 GPU info`,
`M2.4 Bad driver`, `M2.6 GPU vendor cache`, `M3.6 Dependency graph`, `M4.2 SHA-256
integrity`, `M5.2 VC++ Redistributable`, `M5.3 .NET Runtime`, `M5.6 Pending reboot`,
`M6.1 DirectX runtime`, `M6.2 HwSchMode`. Покрытие категорий 1–6 из ТЗ-2 закрыто
на ≈80%.

**Crash bundle export.** Кнопка Export теперь создаёт ZIP с полной папкой краша
(crash_tags + rgl_log + module_list + watchdog + minidump до 100 МБ) + наш
`diagnosis.txt` + двуязычный `READ_ME_FIRST.txt` с инструкцией куда отправить.
Имя файла говорит само за себя: `crashdoctor_unrecognised_<ts>.zip` если краш
не распознан, `crashdoctor_bundle_<ts>.zip` если да. Старый text-only export
оставался бесполезным когда диагнозов 0 — теперь поддержке отправляется всё
сырьё.

**Кнопка «Игнорировать» на Tune-Up карточках.** Persistent ignore-list в
`state/ignored_recommendations.json`, ключ — fingerprint детекта (severity +
summary + sorted evidence lines). При изменении состояния fingerprint меняется,
карточка возвращается на экран. Apply success стирает ignores для модуля.

**Парсер модулей выдерживает ранние крэши.** `CrashCollector.BuildModules` теперь
fallback'ится на парсинг `[Runtime][Arguments][..._MODULES_*A*B*..._MODULES_]` если
секция `Used Modules` отсутствует (Bannerlord не успевает её записать когда падает
на module init). Без fallback все правила с `module_list:` молча промахивались на
этих крэшах — отсюда «не распознан» на конфликтах вроде NavalDLC + TOR_Core.

**iGPU detection через rgl_log.** Новое правило `hw.igpu_actually_selected` матчит
строку `Selected graphics adapter: [N] <name>` — это authoritative источник, а не
порядок устройств в DxDiag. На машинах где DxDiag перечисляет iGPU как
`Display Devices 0`, но игра реально рендерится на дискретке, ложно-позитива
больше не будет.

**Whitelist карт где DxDiag врёт VRAM.** RTX 4070+/4080/4090, RTX 3080 Ti/3090,
RX 7700+/7800+/7900, RX 9070, Arc A770/B580 — все имеют ≥12 ГБ физически, но
DxDiag иногда репортит 3-4 ГБ из-за uint32 saturation в `Win32_VideoController.
AdapterRAM`. `SystemMatcher.MatchGpuField('vram_mb')` для `lt`/`lte` теперь
скипает эти карты (false-positive `hw.gpu_vram_likely_low` на реальном RTX 4080
Laptop был как раз об этом).

- `tor.naval_dlc_conflict` — Naval DLC (War Sails) несовместим с TOR

  CharacterReload/BannerCraft конфликт

  `module_list: { count_above: "5", excluding_official: "true" }` matcher)

**`UnhandledExceptionHandler`.** Регистрируется на `OnSubModuleLoad` ДО любых
наших операций. Слушает `AppDomain.UnhandledException` + `FirstChanceException`,
пишет managed stack в `crashdoctor.log` без swallow'инга. Throttle на 200

## v1.3.12 — .tpac async I/O fault detection (2026-05-04)

Новое правило `assets.tpac_async_read_burst` — детектит классическую сигнатуру
ошибки асинхронного чтения сжатых ассетов Bannerlord. Когда движок не может
прочитать `.tpac` (битый файл / антивирус блокирует / I/O ошибка диска), он
спамит warning `Trying to make partial read on compressed asset data` десятки-
сотни раз перед managed-исключением CLR (`0xE0434352`). У реальных клиентов
с TOR + большими Workshop-модами этот паттерн встречается регулярно.

Триггер — ≥ 100 вхождений partial-read warning в логе. Severity: critical,
confidence: medium (мы знаем сигнатуру, но первопричин 4 — Verify integrity,
переподписка на Workshop, антивирус-исключение, `chkdsk`).

**Bonus fix в матчере:** `LogLineMatcher` теперь корректно понимает дедуп-
суффикс `(×N)`, который вешает `LogNormalizer.DedupConsecutive` на серии
одинаковых строк. Без этого `count_at_least` видел 1 hit вместо N для любого
паттерна с одинаковым текстом — потенциально шатало `gpu.shader_cache_corrupt`
и любые будущие burst-правила.

Также `Trying to make partial read on compressed asset` и `Unable to open
file for asynchronous read` добавлены в `LogNormalizer.SignificantTokens` —
теперь эти строки попадают в `SignificantLogLines`, не только в Last200
fallback.

### Новые Tune-Up модули

**M2.2 GraphicsConfigChanged.** Авто-детектор пользовательских изменений настроек графики.
На старте мода снимает snapshot 35 графических ключей `engine_config.txt` в
`<Documents>/Mount and Blade II Bannerlord/CrashDoctor/state/engine_config_snapshot.txt`.
При следующем открытии Tune-Up — diff с текущим конфигом. Если игрок что-то изменил
в Options → Graphics — карточка появляется с одной кнопкой: очистить кэш шейдеров +
синхронизировать snapshot. После Apply UI явно требует REBOOT + Build Shader Cache,
потому что иначе следующий запуск падает на splash-screen с имперским солдатом
(старый кэш под старые настройки).

**M5.8 HeavyVramApps.** Сканер запущенных процессов на тяжёлых VRAM-консумеров.
33 имени, evidence-based ранжирование:
- **Worst** (1.5+ ГБ даже idle): Oculus / Meta Quest Link (`OVRServer_x64`), SteamVR
  (`vrcompositor`, `vrserver`), Mixed Reality Portal, Virtual Desktop Streamer; Ollama,
  LM Studio; DaVinci Resolve, Premiere Pro, After Effects, Topaz Video AI.
- **High** (0.5–4 ГБ): Blender, Photoshop, Lightroom, OBS Studio, NVIDIA Broadcast / ShadowPlay,
  XSplit, Epic Games Launcher.
- **Medium**: Chrome, Edge, Firefox, Brave, Opera, Discord (HW accel + overlay), Teams,
  Wallpaper Engine.

Если найден хоть один Worst-tier — severity escalates до Critical. Apply открывает
Task Manager (не убиваем сами — VR-runtime cold-kill может залипить шлем до перезагрузки).

**M5.9 LateGameHealth.** Look-ahead диагностика поздней кампании. Читает live
`Campaign.Current` (через TaleWorlds API): день кампании, живых героев, уничтожённых
королевств, активных войн на королевстве игрока. Срабатывает по порогам из
`docs/crash_catalog_2026-05-02.txt`:
- Hero bloat — 800+ → Warning
- Snowball — destroyed/total ≥ 0.4 при day < 200 → Warning
- War cascade — 3+ войн на королевстве игрока → Info
- Long campaign — день 500+ при 600+ героях → Info

Apply открывает в браузере страницу самого релевантного fix-мода на NexusMods:
hero bloat → Heroes Must Die (1164), snowball/war cascade → Diplomacy (832),
long campaign → Death Reduced (2497). Карточка скрыта когда `Campaign.Current == null`
(чистое главное меню без загруженного сейва).

### Изменения в M3.3 EngineConfig

`Apply` теперь:
1. Меняет `terrain_quality` (с backup'ом — как раньше).
2. **Сразу же** удаляет shader cache (`ProgramData/Shaders` + `compressed_shaders_cache.sack` +
   per-module `.sack` файлы → Корзина).
3. `NeedsReboot=true` → reboot-banner на табе.
4. UI-сообщение явно требует **обязательную последовательность**: REBOOT → запустить
   игру → Главное меню → Build Shader Cache → ждать 20–60 мин.

Корень: до v1.3.10 M3.3 менял `terrain_quality`, но не трогал кэш. Кэш под старое
значение → следующий запуск hit a mismatch → краш на splash screen. Юзеры
сообщали «после фикса Crash Doctor игра не запускается».

### Recovery doc

`docs/Recovery_If_Game_Wont_Start.md` — manual инструкция для случая, когда мод уже
не открыть (игра упала на splash). Шаги: удалить engine_config.txt, удалить три места
shader cache, Steam Verify, reboot, Build Shader Cache. Для NVIDIA — fallback на
NVIDIA Optimize. Линкуется в Workshop description в секции «If the game won't launch».

### CrashCollector — читает все rgl_log_errors

Папка краша часто содержит несколько последовательных запусков (ПИД 5136, 8144, 8312…).
Watchdog коррелирует с конкретным PID, но текстовая ошибка движка может быть в
`rgl_log_errors_<другой_PID>.txt`. Раньше мы читали только `rgl_log_<watchdog_PID>.txt`
и пропускали ошибки. Теперь `CrashCollector.BuildContext()` сливает все
`rgl_log_errors_*.txt` в `SignificantLogLines` — правила видят полную картину сессии.

### ElevatedExec — захват stderr

`Verb=runas` + `UseShellExecute=true` (требуется для UAC) запрещают redirect stdout/stderr,
поэтому когда elevated PowerShell скрипт падал и писал
`[Console]::Error.WriteLine($_.Exception.Message)` — сообщение терялось, юзер видел голое
«Elevated helper exited with code 1». Фикс: `RunPowerShell` теперь автоматически инжектит
prelude, перенаправляющий `[Console]::Error` в temp-файл, читает его после
`WaitForExit` и кладёт в `r.ErrorMessage`. Все four call-site (M1.1 Apply+Rollback,
M2.5 Apply+Rollback, M5.5 × 2) теперь показывают точную причину сбоя в UI.

### YAML-аудит как pre-flight gate

В `tests/CrashDoctor.Tests/YamlRuleLoaderTests.cs` добавлен тест
`All_module_data_yaml_files_parse_cleanly` — на каждом запуске прогоняет все
`Mod/CrashDoctor/ModuleData/**/*.yaml` через YamlDotNet и фейлит билд при первом
синтаксическом баге. Поймал скрытый баг в `gpu.yaml` v1.3.2 — escape-последовательности
`\P`, `\S`, `\D`, `\M` в double-quoted YAML строках ломали парсинг ВСЕГО файла gpu.yaml,
все `gpu.*` правила тихо выкидывались. Заменил `\` на `/` в путях (Windows понимает оба).

### Сервисные починки

- **Удаление крашей: «удалилось только 1 из N»**. Junction redirect делает
  `BannerlordCrashesDir → CrashCacheDir` физически одной папкой. Каждая crash-папка
  enumerate'илась дважды — по разным путям. Цикл удалял первую → junction уводил на
  физ-target, вторая итерация silent `continue`. Итого реально удалена половина.
  Фикс в `RecycleBinDeleter.CleanupCrashesFolders` и `M14_DumpCleanup.Detect`:
  skip путей с атрибутом `ReparsePoint` — junction'ные пути перечислены отдельно от
  физ-target'ов.
- **M2.1 Clear shader cache** больше не показывается как «найдено 1» при отсутствии
  крэш-маркеров в логах. Если за последние 30 дней нет shader-OOM/X3004/`DXGI_ERROR_DEVICE_REMOVED` —
  карточка скрыта (`Status=Healthy`). Раньше карточка появлялась всегда с Severity.Info
  и припиской «не обязательно» — нарушало принцип «recommendations only when relevant».
- **YAML escape bug в `gpu.yaml`** (см. выше) — все 6 `gpu.*` правил снова работают.
- **Universal scope** — релиз позиционируется как crash-аналайзер для **любого мода
  Bannerlord** (не только TOR). Правила без TOR-зависимости работают на ваниле и
  любых других модах; TOR-specific правила гейтятся через `module_list: TOR_Core`.

### Совместимость

Bannerlord v1.2.x – v1.3.15. Steam build only. M5.9 LateGameHealth требует
`TaleWorlds.CampaignSystem` — добавлен в csproj reference. Зависимостей от других
модов нет: Harmony, ButterLib, BLSE, MCM не требуются.

### Архитектура

- `IRemediationModule` — интерфейс с `Detect/Preview/Apply/Rollback`. `DetectionResult`
  имеет статус `NotApplicable / Healthy / NeedsAction / AlreadyApplied / Failed`.
- `RemediationContext` — общий блок с `Lang`, путями игры, `BannerlordDocumentsDir`,
  `StateDir`, `BackupsDir`, `BusyTracker`-sink для прогресс-репортинга из модулей.
- `RemediationHistoryStore` — JSON-журнал в
  `<Documents>\Mount and Blade II Bannerlord\CrashDoctor\state\history.json`. Вне
  Workshop folder — Steam re-validation иначе восстанавливает файлы.
- `RemediationHistoryStore.Clear()` — сохраняет неоткатанные reversible-записи (чтобы
  юзер не потерял возможность Rollback).
- Recommendations YAML: `ModuleData/recommendations/m{Id}_*.yaml` — длинные тексты
  (description + 4-5 шагов фикса) для каждого модуля, отделены от C#-кода.

### UAC и progress UI

- `ElevatedExec` — единая обёртка `Verb=runas` для `powershell.exe` / `reg.exe`. Один
  UAC consent dialog на операцию, Bannerlord не нужно elevated, никаких helper-exe в
  составе мода (memory rule).
- `BusyTracker` + `Task.Run` — Apply бежит на ThreadPool, UI читает прогресс в
  `TickFrame()`. Heartbeat-таймер плавно двигает bar даже когда модуль не репортит.
- Полноэкранный progress-overlay: имя модуля, детали («Удаление C:\foo (123 МБ)»),
  зелёный bar с процентами, тёмный backdrop.
- `RemediationFeedback` — единый popup для Apply / Rollback с `onDismissed`-callback
  для re-scan'а после закрытия (M3.2).

### TOR-специфика

  TOR требует 10–50 минут перекомпиляции при следующем запуске.
- Новое правило `gpu.shader_compile_oom`: ловит `out of memory during compilation`,

### Build / publish hygiene

- `dotnet build` пишет напрямую в `steamapps\workshop\content\261550\3717685432\` —
  не в `Mod/CrashDoctor/`. Bannerlord грузит мод именно оттуда.
- Новый MSBuild target `StripPdbFromWorkshop`: удаляет `CrashDoctor.pdb` и legacy
  `Launch_with_backup.bat` из Workshop folder после каждого build (privacy: `.pdb`
  содержит абсолютные пути с именем dev-машины).
- XSD-валидация `SubModule.xml` против vendored BUTR-схем (`docs/butr/SubModule.xsd`)
  перед каждым Build — ловим typos в attribute names при компиляции, не в Bannerlord.
- `SubModule.PurgeOwnPdbFile()` runtime safety-net: на старте мода удаляет `.pdb`
  рядом с DLL для уже опубликованных билдов.
